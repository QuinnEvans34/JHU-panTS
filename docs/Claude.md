# Claude / AI Usage Plan

**Living document — updated weekly.** This file states *how* I intend to use Claude and other AI tools on this project, and which tasks stay manual. It satisfies the proposal's "Agent plan" criterion: AI-assisted tasks are distinguished from manual tasks with specific, realistic examples.

> _Note on filenames: the assignment deliverables list calls this file `Claude.md`, while the rubric refers to an "agent-plan.md." This single file serves both roles. If the instructor wants them split, `agent-plan.md` can be a copy of the plan below and `Claude.md` the living usage log._

---

## Guiding principle

AI accelerates **boilerplate, syntax, and debugging**; I own **every decision that affects correctness, methodology, or interpretation**. In a medical-imaging segmentation project, the dangerous failures are silent (misaligned masks, slice-level data leakage, a metric that looks good but hides zero lesion detection). I review AI output against that risk, not for style.

---

## AI-assisted tasks (with examples)

- **Boilerplate code generation** — MONAI transform pipelines, argument parsing, YAML config loaders, repo scaffolding. _Example: "Write a MONAI `Compose` with LoadImaged, EnsureChannelFirstd, Orientationd(RAS), Spacingd, ScaleIntensityRanged for a [-100,300] HU window."_
- **Debugging** — interpreting CUDA OOM errors, shape mismatches between CT and mask tensors, NaN losses. _Example: pasting a traceback and asking Claude to localize a channel-order bug._
- **Documentation** — drafting README sections, docstrings, this file, and the final report's methods section from my notes.
- **API/library lookup** — how a specific MONAI transform or `sliding_window_inference` parameter behaves, instead of digging through docs manually.
- **Refactoring** — turning a working notebook cell into a clean, config-driven module.
- **Test/sanity scaffolding** — generating the structure of the one-case sanity check and overlay-plotting code.

## Manual / human-owned tasks (with examples)

- **Methodology decisions** — choosing Level 4.5 as the target, patient-level splitting, pos/neg sampling ratio, loss function. I decide these and can defend them.
- **Verifying data correctness** — personally inspecting the sanity-check overlays to confirm masks align with the CT. AI cannot see whether the green outline actually sits on the pancreas.
- **Interpreting results** — deciding whether a lesion Dice of 0.4 is acceptable, and reading failure cases. AI can compute metrics; judging them is mine.
- **Scientific honesty** — the "not a diagnostic system" framing, stating limitations, never overclaiming. Non-negotiable and human-owned.
- **Final architecture/scope calls** — what makes it into Level 4.5 vs. the Level 5 stretch goal.

## Guardrails

- I **review every AI-generated block before committing** — especially anything touching data splits, masks, or metrics.
- I **do not trust AI-generated metric numbers**; I verify against MONAI's own metric outputs and spot-check by hand.
- I log notable AI interactions in `AI-usage.md` each week (what worked, what needed correction).
- I keep the agent context file (`CLAUDE.md` at repo root) updated so AI assistance stays aligned with the project's constraints (config-driven design, no committing raw data, no clinical claims).
